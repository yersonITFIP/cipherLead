DOCUMENTACION DEL PROYECTO - LOGIN TRADICIONAL + 2FA TOTP

Fecha de referencia:
- Marzo 2026

Resumen general:
- Se elimino el flujo activo de OAuth con Microsoft.
- La autenticacion principal ahora es tradicional: correo + contrasena.
- El segundo factor es TOTP de 6 digitos.
- El TOTP es compatible con Microsoft Authenticator, Google Authenticator y apps similares.
- El backend protege rutas con middleware JWT + verificacion de sesion.

Nota:
- La tabla oauth_accounts puede quedar en bases antiguas, pero ya no es usada por el codigo actual.

CAMBIOS EN USERS

La tabla users ahora requiere soporte para contrasena:
- password_hash VARCHAR(255)

El backend intenta agregar esta columna automaticamente al iniciar si todavia no existe.

AUTENTICACION PRINCIPAL

Nuevo flujo de acceso:
1. El usuario se registra con correo, contrasena y nombre visible.
2. El usuario inicia sesion con correo y contrasena.
3. Si no tiene 2FA activo, el backend crea sesion inmediatamente.
4. Si tiene 2FA activo, el backend devuelve un challenge temporal.
5. El frontend pide el codigo de 6 digitos.
6. El backend valida el TOTP y recien ahi crea la sesion final.

RUTAS BACKEND IMPLEMENTADAS

Registro y login:
- POST /auth/register
  Crea una cuenta local nueva.

- POST /auth/login
  Valida correo y contrasena.
  Si 2FA esta activo, devuelve:
  - requiresTwoFactor
  - challengeToken
  - user
  Si 2FA no esta activo, devuelve:
  - token
  - user

Segundo factor TOTP:
- POST /auth/2fa/verify
  Valida el challengeToken y el codigo TOTP.
  Si es correcto, devuelve el token de sesion final.

- GET /auth/2fa/status
  Devuelve si el usuario tiene 2FA activo o configuracion pendiente.

- POST /auth/2fa/setup
  Genera un secreto temporal y devuelve:
  - manualEntryKey
  - otpAuthUrl
  - qrCodeDataUrl

- POST /auth/2fa/enable
  Activa 2FA confirmando el primer codigo de 6 digitos.

- POST /auth/2fa/disable
  Desactiva 2FA validando un codigo TOTP vigente.

Sesion y rutas protegidas:
- GET /auth/me
  Devuelve el perfil del usuario autenticado.

- POST /auth/logout
  Revoca la sesion activa.

- GET /protected/summary
  Ejemplo de ruta protegida por middleware.

Utilidades:
- GET /health/db
  Verifica conexion a MySQL.

- GET /users
  Lista usuarios, protegida por middleware.

MIDDLEWARE DE PROTECCION

La proteccion se hace con JWT + tabla user_sessions:
- El cliente envia Bearer token.
- El backend valida la firma JWT.
- El backend verifica que la sesion siga vigente en user_sessions.
- Si la sesion esta revocada o expirada, rechaza el acceso.

Esto protege rutas como:
- /auth/me
- /auth/logout
- /auth/2fa/status
- /auth/2fa/setup
- /auth/2fa/enable
- /auth/2fa/disable
- /protected/summary
- /users

2FA TOTP DE 6 DIGITOS

Descripcion:
- Usa el estandar TOTP.
- No depende de Azure MFA.
- Es compatible con Microsoft Authenticator como app de codigos.

Librerias usadas:
- otplib
- qrcode

Flujo de activacion de 2FA:
1. Usuario inicia sesion con correo y contrasena.
2. Desde la interfaz elige configurar 2FA.
3. Backend genera pending_secret.
4. Frontend muestra QR y clave manual.
5. Usuario escanea con Microsoft Authenticator o similar.
6. Usuario escribe el codigo de 6 digitos.
7. Backend valida y activa el 2FA.

Flujo de login con 2FA activo:
1. Usuario envia correo y contrasena a /auth/login.
2. Backend valida credenciales.
3. Backend detecta 2FA activo y responde con challengeToken.
4. Frontend muestra formulario para codigo TOTP.
5. Frontend envia codigo a /auth/2fa/verify.
6. Backend valida y crea sesion final.

JWT Y SESIONES

Tipos de token:
- Token de sesion final:
  Permite acceder a rutas protegidas.

- Token de challenge 2FA:
  Solo sirve temporalmente entre el primer factor y el segundo factor.

Datos guardados en user_sessions:
- id de sesion
- user_id
- ip_address
- user_agent
- created_at
- expires_at
- revoked_at

OBSERVACIONES IMPORTANTES

- El backend ya no depende de Microsoft OAuth.
- Microsoft Authenticator se usa solo como app TOTP, no como proveedor OAuth.
- Las cuentas creadas anteriormente por OAuth pueden existir en la base, pero no participan en el flujo nuevo si no tienen password_hash.
- Si necesitas reutilizar una cuenta vieja, debes asignarle una contrasena mediante migracion o recrearla como cuenta local.
